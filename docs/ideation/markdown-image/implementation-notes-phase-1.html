Implementation Notes — Phase 1: Markdown Image

markdown-image · 2026-06-05 · Phase 1

image-widgets — widget position uses openFrom with side: -1

Context
Spec implementation step said "Widget 插入位置：closeTo，side=1". But the spec's case-1 expected projection puts {img} BEFORE the source text: <p>{img}<pending>![x](y)</pending>|</p>. A widget at closeTo with side: 1 renders AFTER the source in DOM order — the opposite of what the cases assert.
Decision
Use Decoration.widget(blockStart + openFrom, …, { side: -1 }). This puts the widget before the source text in DOM order, matching every checkpoint's expected projection.
Alternative
Update the case projections to put {img} AFTER source. Rejected — the cases are the user-visible contract and they're internally consistent; the prose hint in the spec was the inconsistent one.

image scan — priority: 1.5 (not 2 as spec snippet suggests)

Context
Spec sample code sets priority: 2. But emphasis is also 2, so ordering relies on the featureSpecs array position. Image must consume ** in ![**a**](b) BEFORE emphasis scans, otherwise strong marks leak into the alt.
Decision
Use priority: 1.5. Image now runs strictly before emphasis regardless of array order, and still before link (3) and autolink (2.5).
Alternative
Keep priority: 2 and rely on featureSpecs stable sort with image placed before emphasis. Rejected — relying on stable sort + array order is more fragile than an explicit priority gap.

live-image cases — projection has three <pending> spans, not one

Context
Spec prose described the projection as <pending>![x](y)</pending> (a single contiguous pending block). But ProseMirror splits inline decorations around mark boundaries: the image mark on x renders as a <span class="md-image-source">, which forces the surrounding syntax-hint decoration into three sibling spans.
Decision
Each case's expectedProjection uses three sequential <pending> blocks (![, alt-text, ](url)). User-visible appearance is identical (all the same syntax-hint class), and the test asserts what the projector actually emits.
Alternative
Use two delim ranges (open + close) like link, so the alt isn't covered. Rejected — when cursor leaves the source, the alt text would stay visible (not in any delim range), which breaks the "hide source on leave" requirement.

CSS — added .md-image-widget to packages/editor/src/style.css

Context
Spec listed CSS location as apps/playground/src/styles.css, which doesn't exist. The playground re-uses the editor's stylesheet via @rte/editor/style.css.
Decision
Added .md-image-widget { display: block; margin: 1em auto; max-width: 100% } to packages/editor/src/style.css — same file as other editor .md-* classes.
Alternative
Add a new styles.css in the playground. Rejected — the widget is editor behavior, not playground-specific styling.

live-image-empty-url-keeps-source-visible — uses ArrowDown as no-op

Context
The test runner requires at least one keyevent per case. Case 4 (and case 8) need to assert state immediately after setSpecMarkdown with no real input.
Decision
Use a single ArrowDown keyevent. With a single paragraph, the runner's adjacentTextblockPosition returns null and the cursor doesn't move, leaving doc state identical to the post-setSpecMarkdown state.
Alternative
Use ArrowRight. Rejected — at end of paragraph, ArrowRight moves the cursor past the closing token to doc.content.size, putting it outside the visible <p> element.
